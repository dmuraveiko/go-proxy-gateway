# Как будет работать Proxy

Нам нужен сервис-прослойка между внутренними сервисами, NATS и интернетом.

Внутренние сервисы сами решают, куда и какой HTTP-запрос отправить. Proxy только
проверяет, что сервису разрешено им пользоваться, выполняет запрос и возвращает ответ.
Бизнес-логику конкретных интеграций Proxy не знает.

## Обычный HTTP-запрос

Например, Service A хочет узнать статус транзакции во внешнем сервисе.

1. Service A сохраняет запрос у себя в БД и отправляет его в Proxy через NATS.
2. Proxy проверяет подпись и права Service A.
3. Proxy сохраняет запрос в своей БД и сообщает Service A: «запрос принят».
4. Service A сохраняет это подтверждение у себя и подтверждает его Proxy.
5. Только после этого Proxy ставит в своей БД отметку «начинаю HTTP-запрос».
6. Proxy выполняет HTTP-запрос во внешний сервис.
7. Полученный ответ Proxy сначала отправляет Service A, а затем сохраняет у себя.
8. Service A сохраняет ответ и подтверждает получение. После этого операция считается
   полностью завершённой.

Подтверждения нужны потому, что обычный NATS сам по себе не хранит сообщения. Если
сообщение или подтверждение потерялось, сторона отправляет его повторно. У запроса
всегда один и тот же ID, поэтому повторное сообщение не создаёт новый HTTP-запрос.

Для Service A это выглядит как обычный синхронный вызов: отправил запрос и ждёшь
результат. Если нужны параллельные запросы, сервис сам запускает их параллельно. Proxy
не пытается менять или восстанавливать их порядок.

## Что происходит при ошибках

Если БД Proxy недоступна до HTTP-запроса, Proxy ничего наружу не отправляет и не
подтверждает запрос. Service A повторит его позже.

Если Proxy упал до отметки «начинаю HTTP-запрос», запрос после перезапуска можно
безопасно вернуть в очередь.

Если отметка уже стоит, а Proxy упал, мы не можем точно узнать, успел запрос уйти во
внешний сервис или нет. Такой запрос получает статус `unknown` и автоматически второй
раз не отправляется. Это защищает от случайного создания нескольких платежей или
других дублирующих операций.

Если HTTP-ответ получен, но БД Proxy временно недоступна, Proxy сначала пытается
передать ответ Service A. Получается два шанса сохранить ответ: в БД Service A или в
БД Proxy. Если не работает ни одна сторона и Proxy в этот момент упал, восстановить
ответ уже невозможно — операция станет `unknown`.

Если NATS или Service A временно недоступны, сохранённый ответ остаётся в БД Proxy и
будет отправляться повторно до подтверждения Service A. Сам HTTP-запрос при этом больше
не выполняется.

## Повторы HTTP-запросов

По умолчанию Proxy делает одну HTTP-попытку.

Безопасные запросы вроде получения статуса можно повторять, если Service A это явно
разрешил. Запросы, которые что-то создают или меняют, повторяются только когда внешний
сервис поддерживает идемпотентность и Service A передал соответствующий ключ.

На 100% гарантировать ровно одно выполнение чужого HTTP-запроса невозможно: внешний
HTTP-сервис и наша БД не участвуют в одной общей транзакции.

## Данные HTTP-запроса

Service A передаёт адрес, метод, заголовки, тело, таймаут и правила повторов. Proxy не
разбирает и не меняет эти данные, не добавляет авторизацию и не следует по redirect.
Ответ также возвращается без разбора и изменения содержимого.

Используется стандартный Go `net/http`, поэтому техническое представление запроса в
сети может отличаться: например, порядок или регистр заголовков и способ передачи
тела. Сами значения заголовков и тело остаются прежними. Если кому-то нужна полная
побайтовая копия сетевого пакета, для этого понадобится отдельный raw-транспорт.

Proxy не ограничивает список внешних адресов. Разрешённый внутренний сервис может
обратиться к любому адресу, доступному из сети Proxy.

## Доступ и несколько Proxy

Каждый внутренний сервис подписывает сообщения ключом Ed25519. В конфигурации Proxy
указано, каким сервисам разрешено им пользоваться.

Клиент явно выбирает логический Proxy по его ID. У одного логического Proxy может быть
несколько запущенных экземпляров. Они используют общую PostgreSQL, поэтому очередь,
ограничения и состояние запросов у них общие.

## Нагрузка и соединения

Для каждого внешнего сервера можно отдельно настроить количество запросов в секунду и
число одновременно выполняемых запросов. Лишние запросы ждут в очереди. Ограничение
общее для всех экземпляров Proxy.

Соединения с одним сервером по возможности используются повторно, чтобы не создавать
новое TCP/TLS-соединение для каждого запроса.

Сейчас лимит запросов считается по секундам. На границе двух секунд возможен короткий
всплеск. Если конкретному серверу нужна строгая пауза между запросами, это ограничение
нужно будет добавить отдельно.

## Webhook

Service A отправляет Proxy команду на создание webhook и получает публичный URL с
секретным токеном. Этот URL Service A указывает во внешнем сервисе.

Есть два варианта ответа на webhook.

Первый — простой. Proxy сохраняет webhook в БД, сразу возвращает внешнему сервису
заранее настроенный ответ, например `200 OK`, и доставляет событие внутренним сервисам
через NATS.

Второй — ответ формирует Service A. Proxy сохраняет webhook, отправляет его Service A,
ждёт от него статус, заголовки и тело ответа, а затем возвращает этот ответ внешнему
сервису. Если Service A не ответил вовремя, внешний сервис получает `504` и может
повторить webhook.

Один webhook могут слушать несколько внутренних сервисов. Для каждого создаётся
отдельная доставка и отдельное подтверждение. Если один сервис недоступен, это не
мешает остальным получить событие.

## Очистка данных

Завершённые и подтверждённые запросы и webhook удаляются из PostgreSQL через заданное
время. Активные и ещё не доставленные записи не удаляются. Записи со статусом
`unknown` по умолчанию хранятся бессрочно для ручной проверки.

## Что ещё нужно согласовать перед production

- реальные лимиты запросов и размеров данных;
- срок хранения завершённых записей и порядок работы с `unknown`;
- нужна ли строгая пауза между запросами для отдельных внешних серверов;
- нужна ли проверка подписей конкретных webhook-провайдеров прямо в Proxy;
- нужен ли отдельный API для просмотра статуса и отмены запросов;
- нужен ли полный обмен подтверждениями для команд создания и изменения webhook.
