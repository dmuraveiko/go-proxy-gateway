# Логика работы Proxy

Это целевая логика системы с учётом последних ответов и уточнений.

## Назначение

Внутренний сервис передаёт HTTP-запрос через NATS. Proxy проверяет права клиента,
выполняет запрос во внешнюю сеть и возвращает ответ через NATS.

Proxy не знает бизнес-смысл запроса, не добавляет авторизацию и не ограничивает список
внешних адресов.

## Proxy, клиенты и базы данных

Один логический Proxy определяется через `proxy_id` и использует одну PostgreSQL.
Обычный режим работы — один процесс на `proxy_id`. При необходимости HA несколько
физических экземпляров одного `proxy_id` могут одновременно работать с этой БД. Для
клиента и внешнего сервера они остаются одним Proxy. У каждого процесса есть
дополнительный уникальный `instance_id` только для внутренней маршрутизации.

NATS queue group распределяет входящие сообщения между экземплярами. DB leases и
locks не дают двум worker одновременно выполнять одну операцию. Общие host limits
также хранятся в этой БД, поэтому добавление экземпляров не умножает нагрузку на
внешний сервер.

Глобальной блокировки на весь Proxy нет: worker выбирают разные строки через `SKIP
LOCKED`. Основная возможная конкуренция — общий пул PostgreSQL и строки лимитов одного
target host. Поэтому несколько экземпляров не включаются без необходимости; это
резервный HA-режим, влияние которого проверяется нагрузочным тестом.

Разные логические Proxy используют разные `proxy_id` и разные PostgreSQL.

Несколько клиентов могут пользоваться одним Proxy. Клиент явно выбирает `proxy_id` для
каждого запроса. Proxy проверяет Ed25519-подпись и разрешение клиента. NATS subjects и
ACL не позволяют клиентам и Proxy получать чужие сообщения.

Клиентская библиотека содержит готовое PostgreSQL-хранилище. Приложение передаёт DSN
или готовый connection pool, а библиотека создаёт и использует собственные таблицы с
префиксом `natsproxyclient_` по умолчанию. Префикс можно изменить.

Клиенту не нужно реализовывать storage-интерфейс самостоятельно. Интерфейс остаётся
только как дополнительная возможность для другой БД или нестандартного хранения.

## Обычный HTTP-запрос

Например, Service A хочет узнать статус транзакции.

1. Service A сохраняет исходящий запрос в своей БД.
2. Service A отправляет запрос выбранному Proxy через NATS.
3. Proxy проверяет подпись и права Service A.
4. Proxy сохраняет запрос в своей БД.
5. Proxy отправляет Service A подтверждение: «запрос принят».
6. Service A сохраняет подтверждение. Исходный запрос больше не нужно повторять.
7. Service A подтверждает Proxy, что ACK сохранён.
8. Proxy сохраняет этот факт.
9. Proxy записывает в БД отметку «начинаю HTTP».
10. Только после успешной записи Proxy выполняет HTTP-запрос.
11. Proxy получает HTTP-ответ.
12. Proxy сначала отправляет ответ Service A, затем сохраняет его в своей БД.
13. Service A сохраняет ответ и отправляет ACK.
14. Proxy сохраняет ACK и подтверждает его Service A.
15. Service A сохраняет финальное подтверждение. Обмен завершён.

Каждое сообщение имеет стабильный ID. Потерянное сообщение повторяется с тем же ID,
поэтому повтор NATS-сообщения не создаёт новый HTTP-запрос.

## Сбои

Если Proxy не сохранил входящий запрос, он не отправляет ACK и не выполняет HTTP.
Клиент повторит запрос.

Если Proxy упал до отметки «начинаю HTTP», запрос после перезапуска возвращается в
очередь.

Если Proxy упал после этой отметки, невозможно доказать, ушёл ли HTTP наружу. Proxy не
повторяет HTTP и доставляет клиенту ошибку «результат неизвестен».

Если клиент ранее уже сохранил успешный ответ, поздняя ошибка «результат неизвестен»
не заменяет успех. Клиент подтверждает ошибку, но оставляет сохранённый результат.

Если HTTP-ответ получен, а БД Proxy недоступна, Proxy сначала пытается передать ответ
клиенту. Если клиент его сохранил, результат не потерян. Если ответ не сохранила ни
одна сторона и Proxy упал, клиент получит ошибку «результат неизвестен».

Если NATS или клиент временно недоступны, сохранённый результат остаётся в БД Proxy и
повторяется до ACK. HTTP второй раз не выполняется.

## Клиентский API для исходящего HTTP

Клиентская библиотека реализует `http.RoundTripper` — стандартное расширение
`http.Client` в Go.

```go
httpClient := &http.Client{Transport: proxyTransport}
response, err := httpClient.Do(request)
```

Библиотека сама создаёт стабильный ID операции и сохраняет его через storage adapter.
При необходимости приложение сможет передать свой ID через context.

Если context вызывающего кода отменён, `http.Client` возвращает эту ошибку вызывающему
коду. Уже сохранённая операция продолжает протокол в фоне и сохраняет поздний результат
в клиентской БД. После рестарта библиотека автоматически поднимает незавершённые
операции из своих таблиц.

Proxy не восстанавливает порядок параллельных запросов. Одна goroutine работает
последовательно, параллельность приложение создаёт явно.

## HTTP-данные и подписи

Proxy передаёт method, исходную строку URL, значения headers и точные bytes body без
прикладного разбора и изменения. URL не собирается повторно из отдельных параметров:
сохраняются исходные path, escaping и raw query. Ответ возвращается таким же образом.

Например, клиент вычислил HMAC от body и положил подпись в `X-Signature`. Proxy
отправит ровно те же bytes body, поэтому внешний сервис вычислит ту же подпись. Если
подпись считается от URL, Proxy не меняет исходную строку URL или порядок query.

Redirect обрабатывает обычный `http.Client`: получив `3xx`, он при необходимости
создаёт следующий запрос через тот же Transport. Cookies обрабатывает его Cookie Jar.

Proxy сам не добавляет `Accept-Encoding` и не распаковывает response body. Если клиент
сам запросил сжатый ответ, он получает исходные сжатые bytes.

Стандартный `net/http` может менять порядок/регистр headers и wire framing. Это
допустимо: подпись считается от URL/body, а не от порядка заголовков. Побайтовая
идентичность всего TCP-пакета не обещается.

## Retry и неизвестный результат

По умолчанию выполняется одна HTTP-попытка.

Повтор включается клиентом явно. Write-запрос можно повторять только если внешний
сервис поддерживает идемпотентность, а клиент передал idempotency key.

После неизвестного результата HTTP автоматически не повторяется даже для `GET`:
клиент получает ошибку и сам решает, создавать ли новую операцию.

Exactly-once невозможен, потому что внешний HTTP-сервис и PostgreSQL Proxy не участвуют
в одной транзакции.

## Если выбранный Proxy недоступен

Клиент повторяет NATS-сообщение тому же `proxy_id` с ограниченным exponential backoff.
Автоматического переключения на другой Proxy нет: прежний Proxy мог уже выполнить
HTTP, и переключение может создать дубль.

Неподтверждённые сообщения повторяются до ACK или остановки приложения. Активные
операции не удаляются по TTL.

## Callback

Клиент регистрирует callback и получает публичный URL с секретным токеном. Proxy
принимает любой HTTP method и передаёт method, URL, headers и body без бизнес-разбора.

Есть два режима:

- static — после записи callback в БД Proxy возвращает заранее настроенный ответ;
- delegated — Proxy передаёт callback выбранному клиенту и ждёт ответ его
  `http.Handler`.

Клиентская библиотека собирает обычный `http.Request`, вызывает переданный
`http.Handler` и возвращает его status, headers и body через NATS.

До вызова handler клиент сохраняет полученное событие в своей PostgreSQL. После
возврата из handler порядок такой:

1. клиент сразу отправляет ответ в NATS;
2. клиент сохраняет этот же ответ в своей PostgreSQL;
3. пока Proxy не подтвердил получение, клиент повторяет ответ с тем же `delivery_id`;
4. после рестарта клиент берёт сохранённый ответ из БД и отправляет его повторно, не
   вызывая handler заново.

Proxy сначала сохраняет callback-ответ по `event_id` и `delivery_id`, затем отправляет
ACK. Повтор того же ответа безопасен: Proxy узнаёт ID, возвращает ACK ещё раз и не
формирует второй внешний HTTP-ответ. Если с тем же ID пришло другое содержимое, Proxy
отклоняет его как конфликт.

Есть одно технически неустранимое окно: процесс может упасть после возврата handler,
но до успешной публикации в NATS и до записи ответа в клиентскую БД. Ни одна сторона
тогда не может доказать, что handler уже выполнялся. Порядок «сначала NATS, затем БД»
минимизирует это окно. Если handler выполняет собственный side effect, дополнительную
защиту даёт его прикладная дедупликация по ID события провайдера.

Один callback могут слушать несколько клиентов. Каждый получает независимую доставку
и ACK. В delegated mode только один заранее выбранный клиент формирует внешний ответ.

Capability token защищает URL. Проверку provider-specific подписи выполняет внутренний
`http.Handler`, потому что Proxy не знает конкретного provider. Если проверка нужна до
ответа внешнему сервису, используется delegated mode.

Повтор одного HTTP callback считается новой транспортной доставкой. Дедупликацию одного
бизнес-события по provider event ID выполняет внутренний сервис.

Регистрация, изменение и удаление callback выполняются идемпотентными NATS-командами.
Proxy сохраняет команду и её результат в своей БД и повторяет доставку результата до
ACK клиента. Пока вызов активен, клиент повторяет команду с тем же `command_id`.

Незавершённые callback-команды пока не сохраняются в клиентской PostgreSQL. Если
клиент перезапустился до завершения команды, библиотека не продолжит её автоматически:
приложение должно повторно вызвать операцию с прежним `command_id`.

## Ограничение нагрузки

Для каждого внешнего host настраиваются RPS, число одновременных запросов и при
необходимости минимальная пауза. Лишние запросы ждут в очереди конкретного Proxy.

Соединения с одним host по возможности используются повторно. Лимиты хранятся в общей
БД логического `proxy_id`, поэтому соблюдаются всеми его экземплярами. Другие
`proxy_id` имеют свои БД и лимиты.

Размеры request/response/callback, HTTP timeout, backoff и retention задаются
конфигурацией с безопасными значениями по умолчанию. Конкретные production-значения не
меняют архитектуру и выбираются при нагрузочном тестировании.

## Очистка

Завершённые запросы, callback и доставленные ошибки удаляются автоматически после
retention. Активные и ещё не доставленные записи не удаляются.

То же правило действует для встроенных клиентских таблиц: завершённые операции и
обработанные callback очищаются автоматически. После retention прежний ID нельзя
использовать как ключ дедупликации новой операции.

## Разделение кода

Исходящее и входящее направления реализуются отдельно:

```text
client/outgoing.go       http.RoundTripper и исходящие ACK
client/callback.go       доставка callback в http.Handler
client/postgres_store.go durable client storage
internal/httpx/          выполнение внешнего HTTP
internal/app/            входящий callback HTTP
internal/worker/         durable очередь исходящего HTTP
internal/transport/core.go       NATS envelope и маршрутизация
internal/transport/outgoing.go   исходящие HTTP-команды и ACK
internal/transport/callback.go   callback control/events/responses
internal/transport/delivery.go   durable повтор NATS-сообщений
internal/repository/outgoing.go  PostgreSQL исходящего HTTP
internal/repository/webhook.go   PostgreSQL callback
internal/repository/delivery.go  PostgreSQL delivery queue
internal/repository/limits.go    общие host limits
```

Общими остаются только простые типы протокола, подписи, NATS-подключение и работа с БД.

## Зафиксированные уточнения

- Один логический `proxy_id` использует одну БД. По умолчанию это один процесс;
  несколько экземпляров с NATS queue group и DB leases остаются опциональным
  HA-режимом.
- PostgreSQL нужна и Proxy, и клиенту. Клиент получает готовое PostgreSQL-хранилище из
  библиотеки с префиксом таблиц `natsproxyclient_`.
- Исходящий API реализуется через `http.RoundTripper`.
- Callback-клиент вызывает стандартный `http.Handler`.
- Body и исходная строка URL сохраняются без изменения, чтобы подписи оставались
  действительными.
- Порядок headers и точные bytes сетевого пакета не гарантируются, так как используется
  стандартный `net/http`.
